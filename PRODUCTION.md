# Production Readiness Notes

Notes on what needs to change to ship dialin as a multi-user product.
Current state is a personal-use tool with no auth and a manually-entered user ID.

---

## Blockers (must fix before any real users)

### 1. Authentication

**Implemented: optional Clerk.** Set Terraform `clerk_jwt_issuer` (Clerk Frontend API URL, same as JWT `iss`). When non-empty, Lambda verifies `Authorization` session JWTs against that instance’s JWKS (`verify_aud=false` so Clerk’s default tokens work) and sets `ALLOW_CLIENT_USER_ID=false`. API Gateway remains open to the world at the HTTP layer — auth is enforced in Lambda — so tighten CORS and consider edge protection before growth.

**Frontend:** copy `web/dialin-config.example.js` → `web/dialin-config.js` (gitignored) — set `clerkPublishableKey`, or leave it empty and use localStorage key `dialin.clerkPk`. The UI loads `@clerk/clerk-js` and sends `Authorization: Bearer <session token>` on API calls.

**Legacy local mode:** leave `clerk_jwt_issuer` empty; Lambda allows `userId` from body/query again and the manual user id field stays visible.

**Still to tighten for production:** replace wildcard CORS, bake API URL into static hosting, and consider usage plans (see below).

### 2. Bake the API URL into the frontend

The "Set API URL" field is dev scaffolding. For a real product the API URL should be injected at build/deploy time (env var or a `config.js` generated by Terraform output) so users never see it.

### 3. Lock down CORS

`api_gateway.tf` currently has `allow_origins = ["*"]`. Once you have a domain, replace with the exact origin.

---

## Secondary (important before growth)

### 4. Rate limiting

No single user should be able to run unlimited Bedrock calls. Add API Gateway usage plans with per-user throttling (requests/second + monthly quota). Cognito or Clerk identity maps cleanly to an API Gateway usage plan key.

A reasonable starting cap: 100 chat turns/day per user.

### 5. Static hosting

Right now the UI is served with `python -m http.server` locally. For real users:
- S3 bucket + CloudFront distribution
- Terraform resources for both are straightforward additions
- Inject the API URL as a `window.DIALIN_API` global via a `config.js` generated from `terraform output`
- CloudFront can also enforce HTTPS and give you a CDN for free

### 6. Tavily usage tier

The free tier is 1,000 searches/month. One active user doing cafe discovery burns that in a week. Upgrade to the $20/month Basic plan (10k searches) before opening to others. Add a DynamoDB cache for city-level recommendation queries (TTL: 24h) so repeated lookups for the same city don't hit Tavily at all.

---

## What does NOT need to change

- **DynamoDB schema** — the `USER#<id>` partition key design is already correct for multi-tenancy; zero schema migration needed
- **Lambda business logic** — all handlers already scope every query to a user ID; just the source of that ID changes (token vs. client param)
- **API routes** — no structural changes needed
- **Bedrock/tools layer** — fully stateless per-request; scales horizontally without changes

---

## Suggested ship order

1. Set up Clerk (or Cognito) + add JWT authorizer to API Gateway
2. Derive `userId` from JWT in Lambda; remove it from all client payloads
3. Lock CORS to your domain
4. Add S3 + CloudFront hosting via Terraform
5. Upgrade Tavily; add query cache
6. Add per-user rate limiting via API Gateway usage plans
7. Set a higher AWS Budget alert threshold appropriate for your expected user count

## Monetization

### Pricing model

A freemium model with a single paid tier is the right fit for an app this focused.
Keep it simple — one decision for the user to make.

| | Free | Pro ($4.99/mo or $39/yr) |
|---|---|---|
| Brew journal | unlimited | unlimited |
| Coffees / roasters / gear | unlimited | unlimited |
| AI chat turns | 20/month | unlimited (soft cap: 200/day) |
| Cafe recommendations w/ live search | — | ✓ |
| Dial-in advice | — | ✓ |
| Cafe visit tracking | — | ✓ |

**Why this split:** The journal (logging brews, coffees, gear) is useful on its own and builds habit. The AI features are where the real cost is (Bedrock + Tavily). Gating AI on Pro keeps the free tier cheap to run.

**Pricing rationale:** At $4.99/month your AWS cost per paying user is ~$0.50–1.00 (Bedrock + infra share), leaving healthy margin. Annual at $39 ($3.25/mo effective) incentivizes commitment and improves cash flow. Both are well below what a specialty coffee habit costs.

### Stripe integration

**Web (Stripe Checkout):**
- User hits a "Upgrade to Pro" button → your backend creates a Stripe Checkout Session → redirect to Stripe-hosted payment page → Stripe webhook fires `checkout.session.completed` → Lambda marks the user's DynamoDB profile with `{ "plan": "pro", "stripeCustomerId": "...", "planExpiresAt": "..." }`
- Stripe handles the card UI, SCA/3DS, receipts, and renewals
- Add a Stripe Customer Portal link so users can cancel or update their card without you building anything

**What you need:**
- Stripe account (free)
- Two new Lambda routes: `POST /billing/checkout` (create session) and `POST /billing/webhook` (handle events)
- `STRIPE_SECRET_KEY` and `STRIPE_WEBHOOK_SECRET` as Lambda env vars (add to `variables.tf` + `lambda.tf`)
- `stripeCustomerId`, `plan`, `planExpiresAt` fields on the DynamoDB Profile item

**Enforcement in Lambda:** On every `/chat` request, read `plan` from the profile. If `free`, check a `chatTurnsThisMonth` counter (a DynamoDB atomic increment). If over the limit, return a 402 with a message pointing to the upgrade flow.

### iOS support

The current vanilla-JS web app can be wrapped in a WKWebView with minimal effort, but the App Store has strict rules about in-app purchases that affect your revenue significantly.

**The iOS billing problem:**
Apple requires that digital subscriptions sold inside an iOS app use Apple's In-App Purchase (IAP) system, and Apple takes 15–30% of revenue. You cannot link out to a web payment page from within the app for a subscription that unlocks app features.

**The pragmatic path:**

1. **Ship a web app first** (PWA — add a manifest and service worker). Users on iPhone can "Add to Home Screen" and get a near-native experience. No App Store, no Apple cut. This is how many small SaaS products start.

2. **If you want a real iOS app later**, implement StoreKit 2 (Swift) for IAP alongside your Stripe web billing. Your backend needs to handle both receipt sources:
   - Stripe webhook → sets `plan: pro` on DynamoDB profile
   - Apple App Store Server Notifications → same DynamoDB update
   - The user's plan status is the source of truth in your DB; the client just checks it on login

3. **Alternatively, use RevenueCat** — it abstracts both Stripe and StoreKit into one API, handles receipt validation, webhook normalization, and entitlement checking. Saves weeks of billing plumbing at the cost of ~1% of revenue at small scale (free up to $2.5k/month revenue).

**Recommended order:** Launch as a PWA → validate there's paying demand → then invest in a native app if the numbers justify it.
